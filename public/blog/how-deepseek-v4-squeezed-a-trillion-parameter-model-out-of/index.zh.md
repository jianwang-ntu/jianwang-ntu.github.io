# DeepSeek V4 拆解：一支资源紧张的小团队，如何造出能与闭源旗舰掰手腕的万亿参数模型

![How DeepSeek V4 Squeezed a Trillion-Parameter Model Out of Limited Hardware](/images/blog/how-deepseek-v4-squeezed-a-trillion-parameter-model-out-of.png)

视频作者带读者通读了 DeepSeek 新发布的 V4 论文，核心论点是：在算力、芯片和团队规模都远不及 OpenAI 的情况下，DeepSeek 通过一连串精巧的工程权衡，做出了一款能与最强闭源模型对标的开源模型，并把所有方法都写在了论文里。这篇博客把视频里讲到的设计取舍，按"注意力—架构稳定性—训练优化—数据中心调度—训练课程"这条主线重新梳理一遍。

## 模型定位：1.6 万亿参数、100 万 token 上下文

视频指出，V4 Pro 拥有 **1.6 万亿参数**和 **100 万 token 的上下文窗口**——大约相当于 75 万英文词，足以塞进整套《哈利·波特》后再追问某一页的细节，对长时间运行的 agent 也意味着任务中途不会"失忆"。

作者强调，这两个数字单独看都很漂亮，但同时实现却异常困难。视频接下来花了大半篇幅讨论：DeepSeek 在没有顶级 NVIDIA 芯片、团队规模约为 OpenAI 四十分之一的前提下，是怎么硬把这两件事同时做到的。

## 注意力的瓶颈：算不过来，也存不下

视频先解释了 100 万 token 上下文为什么难。Transformer 每读到一个新 token，都要回头和之前的所有 token 比较一次相关性——这是 Google "Attention Is All You Need" 论文奠定的范式。第 10 个 token 比 10 次没问题，第 10 万个 token 就要比 10 万次，到 100 万 token 时计算量呈现"天文级别"。

更糟的是 **KV cache**：模型必须为之前每个 token 存下中间结果，作者把它形容为"一张巨大的查找表"。在百万 token 规模下，这张表会膨胀到数 GB，全都得驻留在昂贵的 GPU 显存里。

作者总结：算力爆炸 + 显存爆炸，就是所有大模型撞上的那堵墙。

## 混合注意力：压缩过去，再忽略掉大部分

作者称，DeepSeek 没有用蛮力堆算力（他们也堆不起），而是换了一个更优雅的问题：**如果模型一开始就不必看所有信息呢？**

V4 提出的"混合注意力架构"由三条并行通路组成。

### CSA：把过去打包成块

**Compressed Sparse Attention（CSA）** 把相邻的 token 按小段（如每 4 个）合并成一个更密集的表示。序列长度立刻缩短为四分之一，比较次数和显存随之下降。

但视频提醒：仅仅压成四分之一，在百万 token 下仍是数十万个块，远远不够。

### Lightning Indexer：像内置搜索引擎一样取关键块

接下来引入**稀疏性**。模型用一种被称为 **Lightning Indexer** 的快速内部机制，对所有压缩块快速打分，只挑出最相关的少数块来参与计算，其余完全跳过。

作者把这一步称为"微妙但深刻的转变"——模型不再追求完整记住一切，而是在合适时刻记住正确的事。

### HCA：把整段历史压成极短摘要

**Heavily Compressed Attention（HCA）** 走更激进的方向：每 128 个 token（约一个段落）压成一个表示。序列被压到极短，模型反而可以"廉价地"看完所有历史，获得宏观视野。

### 滑动窗口：保留最近内容的原始细节

压缩必然丢信息。视频特别强调了这一点：如果半百万 token 之前那段话被压成了一个块，要查具体措辞或某个数字怎么办？

DeepSeek 的回答是再加一条**滑动窗口注意力（Sliding Window Attention）**通路，对最近若干 token（如 128）保持完全无压缩、逐字记录。

作者用一个比喻把三者串起来：像学生备考时，**眼前摊开的几页**是滑动窗口；**早期章节的摘要**是 HCA；**翻看高亮笔记**是 Lightning Indexer 的稀疏检索。三种策略在网络中逐层交错，把"长程检索 / 宏观理解 / 短程精确"三件事同时做了。

### 数字层面的收益

视频援引论文给出的对比：相比已经被业界视为高效代表的 V3.2，V4 Pro 在拥有更大参数量和更长上下文的情况下，FLOPs 仅为前代的约 **27%**（约 3.7 倍效率提升），KV cache 缩减到约 **10%**——短期记忆显存占用直接砍掉九成。

## 万亿参数下的"啸叫"：信号爆炸怎么压住

接下来视频转向第二个挑战：1.6 万亿参数意味着层数极多，信号在层间反复传播容易被放大，作者把这种现象比作"麦克风太靠近音箱发出的高音啸叫"——只是放大的不是声音，而是数字。这就是**信号爆炸（signal explosion）**，是训练崩溃的常见原因。

业界的传统解药是 **residual connection**（残差连接）和后来更宽的 **hyperconnections**。但 DeepSeek 论文明确指出：到了万亿参数规模，连标准 hyperconnections 也会出现灾难性尖峰。

### Manifold Constrained Hyperconnections（MHC）

DeepSeek 把残差通路约束在**双随机矩阵（doubly stochastic matrix）的流形**上：每行加和为 1，每列加和也为 1。作者用更直白的话解释：这意味着信号总量始终守恒，**数学上禁止**它放大。

为了在万亿规模上把这个约束真正强加上去，他们用 **Sinkhorn-Knopp 算法**——在每层处理信号前，对矩阵进行约 20 轮行列归一化迭代。

听上去像在每层插入 20 步循环会拖垮性能。视频指出，DeepSeek 通过 fused GPU kernel、选择性重算等底层优化，把整个开销压到运行时的约 **6.7%**——为防止训练崩溃买的一份廉价保险。

### Muon 优化器

DeepSeek 还把业界默认的 AdamW 换成了自研的 **Muon**。作者把它比作给吉他调音：先大幅、激进地把弦调到接近正确音高，再切换到微调精准对准——"先快后稳"，让模型既学得快又不失稳定。

## 数据中心的瓶颈不是算，而是搬

作者强调，到了 1.6 万亿参数，最大的瓶颈反而不是计算，而是**通信**：模型层不可能塞进一台机器，必须散布在数据中心多个机架间，数据要在机架之间不断迁移。GPU 哪怕等几毫秒，也是被烧掉的真金白银。

DeepSeek 的解决思路是把数据传输**编排成连续小波**。第一波数据一到，GPU 立刻开算；与此同时第二波正在网络上飞来，依此类推，计算和通信完全交叠，网络延迟在宏观上被"消除"，链路和算力都长期处于满负荷。

这种微观调度无法靠通用编程语言写出来。论文提到他们用一种叫 **TileLang** 的语言开发 **fused kernels**——把多个数学操作合成一条 GPU 指令，省掉中间结果反复读写。手写 fused kernel 出错率极高，DeepSeek 用 **Z3 SMT solver** 在数学上 100% 证明 kernel 实现正确，并把这部分代码开源到 GitHub。作者评价：这个规模下哪怕十亿分之一的错误率也会持续静默地腐蚀模型，所以"用数学保证正确"是必须的。

## 训练课程：从短上下文一路扩到百万

V4 Pro 在 **33 万亿 token** 上训练。视频指出，如果一上来就把所有数据砸进去，模型会像被丢给量子物理教材的新手一样彻底崩溃。

DeepSeek 采用渐进式课程：起步只看约 4,000 token 的短序列，让模型先掌握语法和局部结构；待训练稳定后，依次扩展到 16K、64K，直至完整的 100 万 token 窗口——"在大脑还在学怎么思考时，慢慢扩张它的工作记忆"。

### Anticipatory Routing：用"略旧"的快照消除噪声

万亿参数 + 百万 token 训练时，**loss spike**（损失尖峰）几乎不可避免。行业惯例是回滚到上一个存档重跑，又贵又治标不治本。

DeepSeek 的做法被称为 **anticipatory routing**：路由决策不依据"当下"的参数，而依据**略早版本的历史快照**。作者用股票图作类比——日内波动嘈杂，移动均线才显示真正的趋势；用旧快照相当于自动滤掉短期噪声，锁定底层走势。

为了避免一直牺牲实时性，系统会监控自身统计量：在察觉损失尖峰前兆时启用 anticipatory routing，危机过后再把控制权交回实时路由。整个训练因此能"自我稳定"。

## 性能：与闭源旗舰平起平坐

视频展示的对比里，V4 在知识、推理、agent 能力等多个基准上与 Opus 4.6 Max、Gemini 3.1 Pro 等闭源旗舰大致相当；与前代 Claude 旗舰 Opus 4.6 Max 的胜率比较中，V4 平均胜率更高，并在多个任务上反超。

作者还点出两个被强调的成绩：

- **Putnam 2025**：被视为全球最难的本科数学竞赛之一，V4 拿下 **120/120 的满分**。
- **百万 token 极限检索**：当上下文被推到 100 万 token 上限时，V4 的检索准确率甚至超过同样支持百万 token 的 Gemini 3.1 Pro。
- 在 Artificial Analysis 的独立排行榜上，V4 Pro 是仅次于 Kimi 2.6 的开源第二名，已经逼近 Google、Anthropic、OpenAI 的闭源顶尖模型。

## 视频留下的核心观点

作者最后把整套设计串起来作结：交错式注意力解决记忆瓶颈、约束架构压住信号爆炸、Muon 加速学习、底层 kernel 把数据中心榨干、anticipatory routing 让训练自稳——**没有任何单点的奇迹，全部是数十个工程取舍的叠加**。

而真正让他觉得"难以置信"的，不是性能本身，而是 DeepSeek 在资源远不及对手的前提下，**把模型连同论文（包括通常属于闭源实验室"最高机密"的基础设施细节）一起开源**。模型已经在 Hugging Face 上免费下载，硬件够的人甚至可以离线运行。视频以这一句作收：在他看来，这是"一支非常 cracked 的团队"留给行业的礼物。
