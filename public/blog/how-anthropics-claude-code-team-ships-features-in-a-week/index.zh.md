# 在最前沿做产品经理：Anthropic 的 Cat Woo 谈如何在 AI 时代造产品

![How Anthropic's Claude Code Team Ships Features in a Week — and What It Means for the PM Role](/images/blog/how-anthropics-claude-code-team-ships-features-in-a-week.png)

在 Lenny 的播客上，Anthropic Claude Code 与 co-work 产品负责人 Cat Woo 描述了一种与传统 PM 几乎截然相反的工作方式：发布周期从季度压缩到一周甚至一天，团队几乎没有流程，PRD 大多被淘汰，工程师可以从在 Twitter 上看到一条反馈一路独立交付到上线。她认为 AI 让代码变便宜，于是真正稀缺的能力变成了"决定要写什么"。

## Boris 与 Cat：80% 心意相通

Woo 介绍，Claude Code 团队的技术负责人 Boris 主要负责产品愿景——三到六个月后产品应该长什么样、"AGI 满血版"应该是什么形态。她自己的工作则是把今天与那个愿景之间的路径填实，并保证市场、销售、财务、容量等跨职能团队达成一致，从而让一项功能就绪后没有任何阻塞。她形容两人 80% 的判断会自然重合，剩下 20% 由各自更关心的一方主导。

## 候选人为什么"接近的方式不对"

Woo 表示自己面试过数百名 PM，常见的误区是仍然按照 AI 之前的节奏思考。在过去，技术变化以 6 到 12 个月为单位，PM 的工作重心是把跨季度路线图与伙伴团队对齐。如今 Claude Code 的功能交付周期已经从半年缩短到一个月、一周乃至一天，PM 应该把精力放在"如何把一个想法在一周内塞进用户手里"，而不是协调多季度计划。

她列出了在这种节奏下 PM 真正该做的三件事：

- **设清晰目标**。LLM 太通用，反而带来歧义。她举例：要服务的是企业里的专业开发者，要解决的是"权限提示太多导致疲劳"，目标是"安全地走到零权限提示"。这样一句话就排除了大量错误的实现路径。
- **建可复用的发布流程**。Claude Code 的几乎所有功能都先以 "research preview" 形式上线，明确告诉用户这是早期产品，可能不会长期支持，从而把发布的承诺成本压到最低。
- **搭跨职能模板**。工程师在内部 dog food 通过后把功能丢进一个常驻的 launch room，文档、PMM、Devrel 第二天就把发布材料拉齐。她认为搭这种"低摩擦默认路径"恰恰是 PM 的活。

PRD 并未完全消失——对模糊度高的功能或需要重基础设施的多月项目，团队仍会写一页纸说明目标、理想用例与当前的失败模式。但日常协同更多依赖每周覆盖整个团队的指标 readout，以及一份明确写着"我们的关键用户是谁、为什么是他们、我们愿意牺牲什么"的团队原则清单，让任何人都能不被 PM 卡住地做决策。

## 速度从哪里来

被问及 Anthropic 为何能保持这种发布密度，Woo 没有把功劳归给尚处预览的新模型 Mythos。她说团队几个季度前就已经这么快了，关键是流程极简加上文化预期：每个人都被默认有权把一个想法在一周甚至一天内推到线上。

她也直接回应了两件外部争议：

- **Claude Code 源码泄露**。Woo 称这是人为失误，相关 PR 经过两层人工审查仍未拦下，团队已加固流程，当事人继续在 Anthropic 工作。
- **限制 Claude 订阅在第三方 OpenClaude 客户端中使用**。她说团队投入了不少精力寻找最平滑的过渡，最终决定优先支持自家产品和 API，并向所有订阅用户附赠了一些 credits 作为补偿。她坦言这是为了优先服务一方产品所做的权衡。

## PM 这个角色还会存在吗

Woo 不认同"工程师能直接发布所以不再需要 PM"的论调，但她描述的现实是：工程师做 PM 的事，PM 写代码，设计师参与产品决策也提交代码。Claude Code 团队的策略是招"有产品品味的工程师"——许多工程师能独立完成从看到 Twitter 反馈到一周内上线的全流程，几乎不需要 PM 介入。她强调产品品味是稀缺技能，不论候选人来自什么背景，只要展现出这种能力就值得招。

她本人是工程师转 PM，团队里几乎所有 PM 都写过代码，设计师过去也都是前端工程师。她认为工程背景在"未来几个月"特别有用，原因不是模型不够强，而是知道一件事大概要花多少工夫，能直接决定"是辩论它还是花一小时做掉它"。但她预期每隔几个月就会出现一次大的能力跳变，导致最有价值的技能组合反复变化——所以更底层的能力是第一性原理思考、低自我（low ego）地填空，以及"愿意戴很多顶帽子"。

## 人脑还能在哪里继续胜出

Woo 给出的判断很务实：在向超级智能逼近的过程中，人类的相对优势集中在三件事——挑选要做什么、判断做出来的东西好不好、把它推向用户。她补充了第四点：常识与 EQ。一次产品发布有上千个细小环节，模型并不总能感知所有利益相关方的关系、偏好以及合适的沟通渠道，这种隐性知识仍然属于人。

被问到如何在不停加速的环境里保持心态稳定，她说团队偏好招那些能"对着挑战微笑"的人——经历过行业起伏、知道什么给自己充电、能接受发出去的功能不够打磨。她形容很多周日晚上看似 P0 的事情，到周一下午会被新的 P0000 取代，必须狠心排序、按时睡觉。

## 我们也付出了代价

代码变便宜带来的副作用是产品一致性下降。过去代码贵，每个产品的位置都会被精心规划，一个用例对一个产品。现在 Anthropic 经常同时发布几个功能形态相近的能力，让外部用户告诉他们哪个更好。代价是新用户更难判断"完成 X 任务的最佳路径是什么"，整个 agentic 工具生态都让人觉得必须每天刷 Twitter 才不掉队。她提到团队最近上线的 `/powerup` onboarding 流程就是对此的反思——本来团队坚持产品应该自解释，但功能太多后还是补上了引导。

## 让 Anthropic 跑赢的两件事

Woo 把 Anthropic 从落后位置反超的原因归结为两点：

- **统一的使命**。"安全地把 AGI 带给全人类"被放在任何单一产品线之上。她举例：如果 Claude Code 失败但 Anthropic 成功，整个团队也会高兴；很多决策本质上是在两件相互竞争的事中比较"哪一件更服务于使命"，决定后所有人买单。
- **专注**。她区分了"使命"与"专注"——使命让团队愿意为公司目标牺牲自己的 KR，专注让团队愿意不去做那些与使命无关的产品（言下之意是有别于某些"押注社交流"的同行）。

## Claude Code、Desktop、Web/Mobile 与 co-work：怎么分工

Woo 给了清晰的使用区分：

- **Claude Code（终端）**：单次编码任务首选，新功能也最早登陆。
- **Claude Desktop**：适合需要前端的工作，可以一边聊一边在右侧 preview pane 看到网页实时变化；对终端不熟悉的非技术用户也更友好；同时是查看 CLI、Desktop、Web、Mobile 所有 session 的统一控制面。
- **Web 和 Mobile**：在外面散步、出差、坐飞机时把任务踢起来，省得"夹着开机笔记本到处走"。
- **co-work**：当输出不是代码时使用——清空 Slack、收件人收件箱归零、给客户做幻灯片、写一份功能目标文档。

她展示了 co-work 的真实用法：为即将到来的 "Code with Claude" 大会准备演讲，她把 Slack、Google Drive、Calendar、Gmail 全部接上，然后让 co-work 浏览 Twitter、内部 launch room、Claude Code announce 频道，连夜整理出一份 20 页幻灯片，第二天早上她只需做小幅修改。因为接入了 Anthropic 的设计系统，输出"看起来像 Anthropic 设计师做的"。她强调这个例子里 PM 的不可替代之处仍然在最后那一步——读完模型生成的提纲，决定哪些观点真正进入终稿。

她还分享了一位 Claude Code 销售同事自建的内部工具：抓 Salesforce、Gong 和会议纪要，针对每位客户自动生成定制版 deck（如客户用 Bedrock 就抽掉 Claude for Enterprise 的相关页，关心代码审查就加上代码审查页），把原本 20–30 分钟的工作压缩到几秒。Anthropic 内部因 Claude Code 出现了一波"个性化办公软件"的爆发，但 Slack 仍然是公司的事实操作系统——Woo 评价它是"被低估却几乎没人想去重做的核心通讯基础设施"。

## "AGI pill"要嗑得刚好

Woo 认为做 AI native 产品最难的技能，是判断"产品一个月后该长什么样"。最容易踩的坑是嗑得过量——只为想象中无所不能的超级 AGI 模型设计："那其实就是一个文本框，让模型自己决定调什么工具"。难的是为眼下这一代模型设计：怎么把用户引到模型的强项上、怎么用 harness 补它的弱项、怎么让"金路径"被走出来。

她的修炼方法有三：

- **大量与模型对话，并让它反思自己**。比如模型改了前端却不打开 UI 验证，直接问它原因，常常会暴露 system prompt 的歧义、子 agent 没做验证之类的真实漏洞，进而修补 harness。
- **挑出五个最能描述模型好坏的同事**。新模型出来时她会在团队午餐挨个问 vibe，总有几位能说清"这个模型说理太突兀""它太爱写 memory 了，但质量不一定高"——再把这些假设拿到数据上验证。
- **写 evals**。她说不需要写几百个，10 个高质量 evals 就足以让团队对齐目标和差距，是 PM 和工程师都被低估的工作。

她特别点名两位"品味权威"：塑造 Claude 性格的 Amanda——这个任务比代码评测更难，因为没有标准答案；以及整支 Claude Code 团队，靠日常午餐喂回来的实时直觉。

## 模型变强后，要做的常常是删功能

每次新模型上线，团队会重新通读整个 system prompt，逐节问"这条提醒模型还需要吗？"，能删就删。Woo 给了一个具体例子：早期 Claude Code 在做大型重构时常常改了 5 处就停下，团队成员 Sid 借鉴 IDE 的多调用点列表给模型加了 to-do list 工具，把成功率拉了上来；到了 Opus 4 之后，模型自己就会规划列表，工具仍然保留只是因为方便用户旁观，模型是否使用都不再影响完成度。

更让她兴奋的是新模型解锁的全新功能。代码审查就是一个例子——团队此前几次尝试都因为准确率不够而搁置，直到 Opus 4.5/4.6 与 Sonnet 4.6 出现，他们才敢让多个审查 agent 并行扫描整个代码库，把"必须通过 Claude review 才能合并"的内部依赖立起来。Woo 由此推导出一条做产品的姿势：**先做今天还不成立的产品**——这样当下一代模型到来时，把它换进原型，差距就直接被填平。

## 多 agent 时代的产品形态

她把 Claude Code 与 co-work 的演进拆成几块积木：

- **任务**：单次任务能稳定产出可合并、可分享的结果，是基础。
- **多任务**：2025 年下半年起，多 agent 同时编码已成常态。
- **几十乃至上百 agent**：本地内存撑不住，要把它们搬到远端，并设计新的界面让人类能快速判断该看哪个任务。
- **可信完成**：当 agent 说"做完了"，人类要能在很短时间内验证它真的做完了。
- **自我改进**：用户给的反馈要被吸收，下次永远不再犯。

## 给所有职能的一条建议

被问到普通人如何在 AI 浪潮中存活并发光，Woo 的回答是：找到自己每天重复做的、不喜欢的那部分，把它交给 Claude Code 或 co-work 自动化。但她加了一层更严格的标准——**95% 的自动化不算自动化**，最后那 5% 到 10% 才是是否能真的依赖它的分水岭。她坦白自己也在尝试用 co-work 做到 Gmail 收件箱归零，目前还没做到完美。她同时提醒：不要陷入"无止境定制工作流"的另一极端，看到很多人忙着加 skills、MCP、各种增强，反而忘了原本要做的产品。

她引用 Karpathy 的观察解释为何这件事重要：试过 2024 一代基于 chat 的 AI 后失望放弃的人，与正用 Claude Code 写代码的人之间已经形成了认知鸿沟。Woo 的描述是："新一代产品是行动型的"，当人们第一次看到 agent 真的去把事做了，那一刻才会理解什么是 AI 的杠杆。

## "Just do things"

她把自己的工作哲学浓缩成一句"just do things"：很多公司里岗位边界、团队边界划得太死，"这块代码我们碰，那块不能碰"。她说这些边界很多是虚的——只要理解了约束、理解了在为什么优化，就该去推动事情发生，做错了再修、再道歉。她把这种习惯归功于早年在 Scale 只有 20 人时的经历：没有流程、没有岗位说明、只有一堆庞大的问题要解。

聊到 AGI 真的到来后的打算，她预计 AGI 在社会层面扩散仍要很久，自己短期会继续做"把世界带过去"的事；至于之后，她想搬去 Fontainebleau 抱石、把现在每周 0.5 本的阅读量提到一两本，去补物理、机器人、航空航天这些她"明知 AI 已经懂、但仍然想自己懂"的领域。

## 留给读者的话

Woo 反复回到同一条主线：代码变便宜，因此**决定写什么**变成了最稀缺的能力；流程变薄，因此**自己把事做完**变成了最稀缺的习惯；模型变强，因此**为还不成立的产品做原型**变成了最聪明的押注。她希望听众别只一次性玩玩 prompt——挑一件每天都做的小事，把它磨到 100% 可靠，再把节省下来的时间投入到那些"一直想做但没空做"的事情上。她说这才是 AI 真正给到普通人的杠杆。
