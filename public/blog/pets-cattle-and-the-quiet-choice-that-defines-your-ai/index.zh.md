# 当“别把AI agent当宠物”这种话不再悬浮：一次踩坑后的架构反思

![Pets, Cattle, and the Quiet Choice That Defines Your AI Agent's Architecture](/images/blog/pets-cattle-and-the-quiet-choice-that-defines-your-ai.png)

最近 Anthropic 的 Managed Agents 让“AI agent 不应该是被精心呵护的宠物，而应该是日夜工作的牛马”这类说法再次刷屏。一位独立开发者在自己的视频里坦言：这些抽象表述听起来都对，但真要动手做时，反而无从下手。他选择从自己实际踩过的坑出发，用一段 Human-in-the-loop 的实现经历，反推这些“高大上”的概念到底在指什么。

## 起点：Vercel 的 15 分钟封顶

讲者首先描述了把 AI agent 上线网页端时的硬约束：AI agent 是一个长任务，可能跑 20–30 分钟，而 Vercel 这类 Serverless 后端的 HTTP 阻塞时长上限大约是 15 分钟，几乎所有 Serverless 厂商都有类似限制。这就决定了不能用一个阻塞的 HTTP 调用把任务跑完，必须改成异步：

- 前端发起“启动服务”请求；
- 任务在另一个云端后端运行，返回一个服务 ID；
- 前端凭 ID 去轮询任务状态。

讲者强调，这只是前后端分离的标准做法，并不新鲜。

## 一个看似不起眼的决定：数据库作为唯一真相

既然 HTTP 通道断了，状态就必须落到别处。讲者把**和 AI agent 交互的所有信息——用户消息、AI 输出、agent 每一步的执行轨迹——都写进数据库**，前端再从数据库读出来显示。

他特别提到一个例外：如果要做对话流式显示（Streaming），就不该用数据库，而应该用 Redis 或者干脆用现成的 pub/sub 服务，他自己用的是 Apple 提供的对应能力。

讲者承认，他当时并没意识到“把所有交互作为 Source of Truth 存下来”这个决定有多重要。后面架构上的几乎所有事情，都是从这一个小决定衍生出来的。

## Human-in-the-loop 的两条路

接着出现了真正复杂的需求：用户要能在 agent 运行中途介入——回答问题、终止某一步、改完再让它继续。讲者指出这件事有两种解法。

### 解法一：挂起进程，等用户输入

agent 运行到关键步骤时，进程直接挂起，停在原地等用户输入，输入到达后再继续往下跑。已有一些 SDK 在朝这个方向做。

### 解法二：直接结束，再开新一轮

讲者当时为了省事选了另一种：**到了需要用户输入的步骤，根本不等待，直接结束这次调用**。等用户给出选择后，把之前存下来的全部上下文加上新输入，作为一个新的 API call 发给 AI agent，相当于发起新一轮对话。他甚至当时都不觉得这是一种“刻意选择”——既然已经把所有用户消息和 agent 步骤都存进数据库，最自然的实现就是停下，让用户敲“继续执行”，然后再跑。

## 一次自我怀疑：Durable Execution 是不是更对？

做完这版功能后，讲者去读 Inngest、Trigger.dev、Temporal、以及 Vercel 最新 WDK 的文档，发现它们都在主打**Durable Execution**：把 agent 的每一小步都当作可恢复的 step 来跑，崩了能自动恢复，并且原生提供 `Wait for Approval`、`Wait for Token` 这类“等待事件”的语义。

这让他慌了两件事：

1. 他写 agent 时**完全没有 Durable Execution 的概念**——他只是把 Inngest 当作一个“能跑得久的 Lambda”，整个 agent 一股脑塞进去，绕开 Vercel 的 15 分钟限制；
2. 他的实现“需要人手动说继续才会继续”，看起来朴素、土气，似乎根本没用上这些云服务的潜力。

他一度纠结要不要重构去贴合那套框架。

## 调研结论：这是两种血缘，不是先进与落后

讲者忍住了重构的冲动，转去研究为什么两种范式可以“殊途同归”。结论是：**两种解法的区别来自它们的起源**。

- **Wait for token 这一派源自传统工作流**，Temporal 的文章是典型代表。工作流走到需要人工介入那一步，就该原地等，可能等一天、三天，配合超时机制。这种“等待”机制在 AI 出现前就存在于传统 workflow 里。
- **直接结束、再开新一轮这一派源自 AI chatbot**。对于一个对话式 agent，用户输入和模型输出统一被看作 message 列表里的一项，下一步生成完全由前面的对话内容决定。停下来等用户其实是**会话退出了进程**，新输入到来后再据此生成下一段——本身就是离散的，没有“持续运行的进程”这个概念。

业界把前者叫做 **Stateful Human-in-the-loop**，后者叫做 **Stateless Human-in-the-loop**。讲者发现，自己用的“土方法”其实是 Stateless 派的标准玩法。他还观察到，Claude Code 等主流 AI agent 编程工具，并没有真正用 durable execution 那一套，而是把全部对话作为 source of truth，把 agent 看成一步步离散的运行——而 Inngest、Temporal 之所以推 durable execution，是因为它们最初服务的是传统工作流，不是 AI。

## 这才是“宠物 vs 牛马”的真正含义

绕了一圈，讲者再回头看 Managed Agents 那篇文章里被反复引用的话——“AI agent 不应该是宠物，应该是牛马”，他说自己突然懂了。

在他自己的架构里，**只要数据库维护好，任何一个执行单元只要能读到数据库就能继续**。不需要知道某个进程的 PID，不需要保证某个进程一直在线。讲者认为，这正是文章里说的 Session 与 Harness 的分离。

反过来，Inngest、WDK 等服务在 Wait for token 模式下会返回一个 `run id`，那个 run 才是必须被精心呵护、不能掉线的“宠物”。而牛马是无差别的：哪一匹来都行，能干活就行。

讲者承认，要不是亲手踩过这套，他即使读过那篇文章，也很难把“宠物/牛马”“Session/Harness/Sandbox”这些抽象词与自己面前的技术选型对上号。

## 留给读者的判断

讲者预测，**Stateless 这种形态在未来可能成为主流**，Durable Execution 的存在感会越来越薄；除非面对的是一个非常确定的工作流，否则没必要按 Inngest、Temporal 教程里那样去精心编排 AI agent。

他也提醒了一个容易踩的心理陷阱：当看到成熟商业方案“官方支持 Wait for Token”时，本能会觉得对方更高级、自己只是临时凑合，从而想去重构。但这两种方案其实只是血缘不同，并无高下；选型时真正该问的是“在我现有的技术条件下，哪一种更顺手”。

讲者最后给出的态度并不浪漫：AI agent 的开发不是把流行的术语和架构堆起来就够了，关键是把那些看起来高大上的抽象概念，和实际开发里遇到的接地气的问题对应起来——只有这样，才能做出真正适合自己产品的决定。
