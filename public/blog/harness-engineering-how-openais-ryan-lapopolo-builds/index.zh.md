# 代码已经免费：Ryan Lapopolo 谈如何让代理"做完整的工作"

![代码已经免费：Ryan Lapopolo 谈如何让代理"做完整的工作"](/images/blog/harness-engineering-how-openais-ryan-lapopolo-builds.png)

在伦敦一场以"Harness 工程"为题的演讲中，OpenAI 技术员工 Ryan Lapopolo 抛出了一个挑衅性的论断：实现层面的代码不再是软件工程的稀缺资源——稀缺的是人类的时间和注意力，以及模型的上下文窗口。他把过去九个月称为自己"完全用代理写代码"的实验，并禁止团队成员触碰编辑器，所有工作必须经由模型完成。

## 中心论点：代码不再是瓶颈

Lapopolo 自称是"Token 亿万富翁"，每天消耗超过十亿输出 token，每天约一千美元的开销。他认为，2025 年下半年，三件事汇合起来改变了软件工程的形态：模型能力（他把 GPT 5.2 视为转折点，这一代模型在他看来已经"和工程师同构"）、harness 的成熟、以及代理在长时间任务上的可靠性提升。

他主张，过去工程师把代码当作负担，是因为它消耗人的同步注意力；但模型"耐心、可无限并行"，因此生产、重构、删除代码的成本几乎归零。在他的描述中，今天每位工程师身边都有"50 到 5000 名工程师等价的算力"全天候待命，瓶颈只剩 GPU 容量和 token 预算。

> 演讲者声称的不是代码会消失，而是写代码不再是工程的难点；难点变成了如何高产地调度这些代理。

## 三种新的稀缺资源

Lapopolo 把工程师的角色重定义为"幕僚长（staff engineer）"——一个人指挥尽可能多的并行代理。在这种模式下，他认为只有三类资源真正稀缺：

- **人类时间**
- **人类与模型的注意力**
- **模型的上下文窗口**

P0/P2 这种排序在过去把 P3 永久压在队列底部；他声称在代码免费的世界里，P3 任务可以同时启动四份并行实现，挑一个能解决问题的合并即可。他举了一个内部例子：在 OpenAI 给同事做的工具，从第一天就支持伦敦、都柏林、巴黎、布鲁塞尔、苏黎世、慕尼黑同事的本地化，因为本地化不再需要从其他团队那里"抢"工程时间。

## 把"好代码"的隐含规则写下来

Lapopolo 反复强调一个观点：完成一次合格的补丁包含约 500 个关于"非功能需求"的小决策，而模型在训练中见过这些决策的所有可能组合。所以工程师真正要做的事，是**把"好"的标准明确写下来**，让代理可以在执行时被这些规则约束。

他列举了团队用来"提示注入"代理的若干工具：

- `agents.md` 与各种角色化的文档（前端架构、后端可扩展性、产品视角等）
- 在 CI 中每次推送都跑的安全与可靠性 review 代理，比如检查 `fetch` 调用是否包了超时和重试
- 针对源代码本身写的测试，比如**限制单文件不超过 350 行**，目的是为了模型上下文友好
- 给出可执行的修复提示的 lint 错误信息，而不是只报错
- 嵌入在测试里、用 Agent SDK 调用的代理来评审代码

他自嘲说，自己作为人类是"不可靠的网络代码作者"——会修一次超时和重试就把这件事忘掉。但写一个 lint 规则把整个仓库迁移过来，就一次性把这个隐患永久消灭。"大规模重构现在是免费的，所以**让所有东西尽量一致**是你能做到的。"

## 一切皆 prompt

演讲中段，他抛出一个观察："Powers 是 prompt，rules 文件是 prompt，skills 是 prompt，lint 错误信息是 prompt，review 代理在 PR 上的评论也是 prompt。"在他看来，这一切都不需要碰模型权重，只是把指令在恰当的时机塞到代理面前。

他还演示了一种"嵌套式"的杠杆使用方式：把 OpenAI 开发者文档里的 prompt cookbook 喂给 Codex，让它合成一个"如何写 prompt"的 skill；之后他自己写 prompt 时再调用这个 skill。"我用代理写出来的 prompt，去看代理写 prompt，再写 prompt。"

## Harness 的设计哲学：刚刚好的上下文

被问到如何避免过度工程化 harness 时，他援引了"苦涩的教训"：不要做会被下一代模型直接淘汰掉的工作。他认为上下文管理是不会过时的——模型永远需要在合适的时机被告知任务要求和约束。

他主张 harness 的全部职责就是**在正确的时间把正确的文本送到模型面前**。例如：React 组件应当被拆得足够小、状态足够少这种要求，不应该在任务一开始就塞进上下文；正确的做法是让代理先"做菜"、做出原型，等到 lint 或 test 阶段再用提示告诉它"现在请按照下面的标准重组"。

## 工作流与团队实践

在与主持人的对谈中，Lapopolo 描述了他和团队的实际做法：

- **入口是 Codex，不是 shell。** 应用、本地可观测性栈、Chrome DevTools 的连接，全部封装为代理可调用的 skill。
- **5–10 个高质量 skill，而不是上百个。** 他更倾向于把已有的 skill 做深，因为仓库里的本地工具变化太快，他没有时间去追。
- **Monorepo 强结构。** 他从一个空 Electron 仓库起步，最后演化为 PNPM workspace 中的约 750 个包，按业务域和技术分层切分，目的是让代理在任何子目录里都能看到一致的代码风格、一份 ORM、一种并发助手、一种 CI 脚本写法。
- **markdown 文件 + GitHub PR 作为协作中心。** PR 是"广播域"，人类和代理都可以评论，实现代理可以承认、推迟或拒绝任何反馈——他说这避免了"代理被一群评审者霸凌"的失败模式。
- **周五是"垃圾回收日"。** 团队成员把这一周观察到的所有 slop 收集起来，找方法系统性消除——把人类的同步评审，逐步转化为仓库里的文档、自动化的 lint、按角色触发的 review 代理。

他还顺带提到一个画面：把笔记本电脑用安全带固定在车后座，开着 Codex 通勤。"理想状态是 50 个代理 24 小时不停地跑，而我根本不需要和它们交互。每次我得手动输入 continue，都是 harness 没把'继续到完成'的上下文交代清楚。"

## Token 预算花在哪里

被问到他每天那十亿 token 的去向，他粗略估计是大约三等分：**计划与工单梳理 / 实现 / 在 CI 中跑的检查**。他不太用 plan mode——他认为如果不会逐行读计划就批准，等于把一堆"可能错误的指令"提交给代理执行，反而浪费 token。如果一定要做计划，他建议把计划本身作为单独的 PR 走人工评审。

## 把代码当作可丢弃的产物

当观众问"代码是不是一种可丢弃的构建产物"时，他直接回答"是"。他借用编译器作类比：harness 工程里写下的所有规则与文档，相当于编译器后端的静态分析与优化 pass；模型像一个"模糊编译器"，换模型就像把 LLVM 换成 Cranelift——只要前端约束足够明确，后端生成的指令可以不一样，但都应是合规的。

## 演讲落幕处的图景

Lapopolo 给出的"未来"画面是：拿到一个 token 预算和一个季度的优先级排名，把成功指标和可靠性指标交给机器，剩下的事不再需要人手在方向盘上。在他描述的这个工作模式里，工程师真正花时间的地方，是排查用户反馈、值班响应、读 Twitter 上的口碑、为运维写好可执行的 runbook，再把这些 runbook 转化成系统约束——所谓"用代理给代理写规则"的元编程层。

按照他的总结：代码已经免费，不要犹豫把自己从循环里拿出来。代理能做完整的工作。
