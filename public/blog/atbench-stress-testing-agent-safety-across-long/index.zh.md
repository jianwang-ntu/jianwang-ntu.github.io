# ATBench：用 1000 条长轨迹给智能体安全评测“补课”

![ATBench: Stress-Testing Agent Safety Across Long Trajectories, Not Single Prompts](/images/blog/atbench-stress-testing-agent-safety-across-long.png)

LLM 智能体的安全风险往往不在单条回复里，而在多步交互的链条上累积。这篇论文提出 **ATBench**——一个轨迹级智能体安全基准，从 *风险来源、失败模式、现实危害* 三个正交维度来组织风险，并通过分类指引下的合成与人工全审来兼顾多样性与真实性。其结果是：哪怕是 GPT‑5.4，在二分类安全判定上也只拿到 76.7% F1，在风险来源细粒度归因上更是只有 33.6%。

## 现有轨迹级评测的三个短板

作者把已有基准（R‑Judge、AgentSafetyBench、ASSEBench 等）的不足归纳为三点：

- **交互多样性不足**：工具池窄、场景模板化；
- **失败可观测性差**：粗粒度 safe/unsafe 标签掩盖了风险如何产生与演化；
- **缺乏长程真实性**：轨迹太短，难以复现“先布点、后引爆”的延迟触发风险。

ATBench 的设计目标就是“在保持真实性的前提下最大化多样性”。

## 三维安全分类法作为构造脚手架

论文把不安全轨迹沿三个轴拆解：

- **风险来源（8 类）**：用户输入（恶意指令/越狱、直接 prompt injection）、环境观测（间接 prompt injection、不可靠/错误信息）、外部实体（工具描述注入、恶意工具执行、被污染的工具反馈）、智能体内部失败。
- **失败模式（14 类）**：分为行为类（越权动作、规划失误、工具选用/参数/上下文使用错误、未验证工具输出、不安全交互、流程偏离、浪费式执行）与输出内容类（违规内容、违法指引、恶意可执行文件、未授权信息披露、误导信息）。
- **现实危害（10 类）**：隐私、财务、系统完整性、人身健康、心理、声誉、信息生态/社会、公共服务、公平/分配、功能/机会损失。

每条不安全轨迹按 *因果先后* 只打一个主标签：先定位最早的风险来源，再选主导失败模式，最后选主要后果。这种“单主标签”策略让分层评测更稳定。

## 一条轨迹是怎么造出来的

构造引擎以分类法为采样空间，配上一个由 RapidAPI、ToolBench、ToolAlpaca 以及补足覆盖的模拟工具拼成的 **异质工具池**（2,084 个可见工具）。Planner 先给每个样本生成一份蓝图——用户任务、候选工具、步骤、触发位置、目标安全结果——再由 Query Generator、Tool Description Simulator、Tool Call Simulator、Simulated Data Generator、Agent Response Generator 串行执行。

值得注意的两点：

- **配对的安全/不安全变体**：同一骨架下，不安全版本让风险落地，安全版本则让智能体拒绝、验证或正确处理。
- **长上下文延迟触发协议**：把轨迹拆成 SETUP 与 EXPLOIT 两个连续 episode，前一段建立的状态、权限、工具输出会被后一段直接复用——也就是说，触发点显式标记为 `source=CONTEXT`，要从早期上下文里“接水”，而不是临时塞进一个本地触发器。这比单纯加长轮次更能反映“早布点，晚爆雷”的真实风险。

数据质量则靠三道关：基于规则的格式/schema 校验、基于 LLM 的合理性筛查、以及五名审稿人对全部 1,000 条轨迹的人工通审。审计共修正 5 条二分类标签和 165 条细粒度标签（覆盖 129 条不安全轨迹），最终发布 503 safe / 497 unsafe。

## 数字上 ATBench 比同类更“长”

与 R‑Judge、ASSE‑Safety/Security、以及 ATBench 早期 500 条版本相比，ATBench 的可见工具数（2,084）、调用工具数（1,954）、平均轮次（9.01）都更高，平均 token 数从 1.52k 跃升到 3.95k——表 1 显示这是相对早期版本 +159.9% 的增长，主要来自延迟触发的长上下文设计。

## 实验：识别“出问题”容易，归因“怎么出”很难

论文在前沿闭源模型、开源通用模型、以及专用 guard 模型三类评测者上跑了实验：

- **二分类（ATBench‑C）**：GPT‑5.4 76.7% F1，Gemini‑3.1‑Pro 75.0%，AgentDoG‑Qwen3‑4B 71.1%。Qwen3.5‑397B‑A17B 只有 67.8%，Llama3.1‑8B‑Instruct 高召回但 F1 只到 61.9%。
- **细粒度诊断（ATBench‑F，仅在不安全子集上）**：GPT‑5.4 在 Risk Source / Failure Mode / Real‑world Harm 三轴上分别为 33.6% / 13.5% / 30.2%；轨迹监督训练过的 AgentDoG‑Qwen3‑4B 反而最强（46.8% / 16.5% / 40.6%）。多数现成 guard 模型（LlamaGuard、Qwen3‑Guard）几乎不支持结构化归因，召回也很低（Qwen3‑Guard 召回 0.4%）。
- **跨基准难度对比**：同一批模型在 R‑Judge / ASSE‑Safety 上常在 80‑90% 区间，到了 ATBench 大多明显回落——GPT‑5.2 从 90.8% 降到 69.0%、Qwen3‑235B 从 85.1% 降到 59.2%、AgentDoG‑Qwen3‑4B 从 91.8% 降到 64.0%。

按叶节点细看，论文指出难点集中在 *工具相关* 的归因：

- 工具描述注入、恶意工具执行、被污染工具反馈这几类风险来源对所有评测者都很难，许多模型只是把它们错误地重新归类为更熟悉的“用户输入”或“误信息”；
- “选错恶意工具”这一失败模式上 GPT‑5.4 仅 16.7%、Qwen3.5‑397B 22.2%、AgentDoG‑Qwen3‑4B 仅 5.6%——模型能看出结果不对，却说不清机制；
- 在危害维度上，隐私、财务、系统完整性这类“具象化”危害相对好判，而心理/情感伤害（0–16%）、功能/机会损失（< 30%）等弥散性后果则普遍难辨。

## 这项工作不声称什么

作者也明确了基准的局限：每条不安全轨迹只打一个主标签，可能遗漏多因后果中的次要解读；全部轨迹为英文，未覆盖多语言部署；只覆盖文本+工具调用，多模态与具身场景留作后续。换言之，ATBench 展示的是 *评测难度*，而不是 *现实风险全景*——它是给 guard 模型一把更难的尺子，而不是一个完整的运行时防线。

## 一句话外卖

对做智能体安全的研究者而言，ATBench 把“能不能识别出不安全”和“能不能解释清楚为什么不安全”分成两个独立任务来量化——后一个任务，目前还远没有被解决。
