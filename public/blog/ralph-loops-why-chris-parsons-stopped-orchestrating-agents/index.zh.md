# Ralph Loop：把 AI 当成"同一个工程师"无限循环跑下去

![Ralph Loops: Why Chris Parsons Stopped Orchestrating Agents and Started Looping Them](/images/blog/ralph-loops-why-chris-parsons-stopped-orchestrating-agents.png)

在 AI Engineer 大会的一场两小时工作坊里，资深 CTO、连续创业者 Chris Parsons 提出一个看似简单到荒谬的主张：与其费劲编排多代理工作流，不如让 AI 在一个循环里反复挑下一个最重要的任务来做。他认为，这是大多数知识工作未来的真实形态——从写代码、写新闻邮件，到运营整个创业公司。

## 名字的来由：辛普森一家里的 Ralph Wigum

Parsons 解释，"Ralph Loop"得名于辛普森一家中的角色 Ralph Wigum——一个反复尝试同一件事直到成功的孩子。这个想法最早由 Geoffrey Huntley 在 AI 世界的"远古时代"（也就是大约去年六月）提出：每当 AI 完成一项任务，就用完全相同的提示再让它跑一次，看看它会不会发现自己漏掉了什么。

按照 Parsons 的描述，早期的 Ralph Loop 只是这样朴素地工作：

- 让 AI 实现某个功能
- AI 说"做完了"
- 把同一句提示再扔回去
- AI 经常会回头看一眼，说"哦，原来还有一处没改"
- 直到它真的没东西可改为止

他指出，最新一代模型（GPT-5.8 之后的版本、Claude Opus 4.6 或 Sonnet 4.6 以上）已经很少在这个最初级循环里发现遗漏，因此 Ralph Loop 的真正价值已经不在这里。

## 从复杂工作流到一个循环

为了说明思路上的转变，Parsons 展示了他过去用 n8n 搭建的每周新闻邮件工作流：节点繁多、依赖复杂，每周一下午两点准时推送"工作流失败"通知。他坦言，维护那条流水线比自己亲手写新闻邮件还累。

后来，他把整个 n8n 流程的 JSON 复制粘贴给 Claude Code，让它据此生成一个 skill。如今他只需要一句提示，Claude Code 就会在内部循环——读 skill、调工具、再读、再调——直到产出一份比从前更好的初稿。Parsons 强调一个观察：**Claude Code 本身就是一个循环**。所以与其外部精心编排，不如让模型在循环里自己决定下一步。

## 跳出"瀑布陷阱"：让 AI 自己挑下一张票

Parsons 承认自己最初把 Ralph Loop 玩砸过。他曾尝试让 Claude 把一个大项目拆成大量带依赖关系的子票，再启动六七个并行 agent，结果场面一片混乱：多个 agent 同时认为某张共享票阻塞了自己，于是各自把它实现了一遍。他事后写了篇相当沮丧的博客，结论是"无法编排大量 agent"。

但他随后意识到，自己其实是把九十年代最糟糕的瀑布流程交给了 AI 去执行——用厚重的需求文档把整个项目锁死。换一种做法效果立刻不同：

> 不要预先排好顺序，只告诉 AI："挑下一张最重要的票去做。"

按照他的说法，AI 完全有能力扫一眼整个 backlog，根据已完成的部分动态推断依赖，再决定下一步。它不擅长的事情是并行管理这一切——但单线程跑就够了，瓶颈通常不是 token，而是人跟不上 AI 的产出速度。

## 现场演示：Pomodoro 计时器与 docs/tickets

Parsons 在台上用一个三分钟"vibe coded"出来的 Python Pomodoro 计时器作为玩具代码库，配了一个 `docs/tickets/` 文件夹，里面是一组待办：新增 status 命令、stop 命令、自定义时长等。他给 Claude Code 的提示极简：

> "Implement the next most important ticket using TDD principles from docs/tickets. Commit when done."

Claude 自行读完所有票，挑出第二张作为下一项，按 TDD 写测试、改实现、把票标记为完成。最朴素的循环则是一行 shell：

```
while true; do claude -p "implement the next ticket"; done
```

他指出，这个朴素脚本想要不卡住地连续跑，必须放宽 Claude 的权限设置，因此沙箱化是认真使用 Ralph Loop 的前提。

## 一切工作都是循环

Parsons 列出了自己生活中持续运行的几个 loop：

- **心跳 loop**：每 15 分钟检查日历、邮件，必要时发 Telegram 通知
- **早晨简报 loop**：每天早上 6 点生成当天行动指南
- **工人 loop**：在他自建的看板应用里，自动推进每个项目的下一步
- **新闻邮件 skill**：取代了原来的 n8n 工作流
- **创业 skill**（实验中）：试图让 AI 推进整个产品和公司的下一步

他承认创业那条还跑得很糟糕，但已经会自作主张做出投资人更新材料的幻灯片。他特别强调，这种做法很快会逼人面对一个不舒服的问题：**哪些工作其实并不需要我做？**

Parsons 自己的划分是："这件事如果搞砸了，是不是会让我尴尬到无法挽回？"——能可逆，AI 就动手做；不可逆（发邮件、发 LinkedIn、给客户回消息），只允许它准备草稿，最后由人确认。

## skill 的角色：循环的"操作手册"

在 Parsons 的体系里，Ralph Loop 的真正能力来自背后的 skill 文件。他展示了自己的 ralph skill，里面对每一种情况都有详细规则：

- 工程师角色定位（"你是接力队中的一员，做一件事就停"）
- ticket 格式与状态机
- 启动前检查 git 是否干净
- 崩溃恢复：脏工作区 + 测试通过 = 大概做完了，但要再确认
- 测试通过不等于功能正确，必须验证实际行为

他认为 ticket 系统不必复杂——纯 markdown 文件就行；想升级可以试试 Steve Yegge 的 beads，或接到 Linear、Jira。但 skill 这一层必须随项目长期演化，把"我们团队怎么干活"沉淀进去。

## 子代理与对抗式审查

观众分享了一个普遍经验：把验证步骤放在同一个上下文里，AI 会自我安慰说"刚才检查过了，没问题"；改用 sub-agent 验证时，子代理由于上下文干净，能挑出真问题。Parsons 认同这一点，并提到 Anthropic 自带的 simplify skill——它会启动三个子代理，分别检查最近改动是否可简化、是否重复，效果显著。

另一位观众提到的"对抗式审查"（开发 agent 写完后，由审查 agent 反向挑刺再回写）在 Parsons 看来同样有效，但他坦承：**真正的瓶颈最终还是人在审 diff 这一步**。

## 关于 spec 驱动开发的保留意见

被问到是否使用 BMAD method、OpenSpec、Kira 这类 spec 驱动框架时，Parsons 表态偏谨慎。他担心两件事：

1. 把整个项目过度规格化，等于把瀑布流程换皮搬回来
2. 把当前模型的工作方式硬编码进工具，等模型迭代到 Mythos（Anthropic 据传中的下一代）这一切设计可能都过时了

他更倾向于"即时 spec"：临场和 Claude 一起做计划，执行完丢掉，让 skill 文件持续承载长期知识。

## 安全与"致命三要素"

谈到沙箱时，Parsons 引用 Simon Willison 提出的 **lethal trifecta**：当一个 agent 同时拥有不可信输入、互联网访问、可访问的敏感数据，数据泄露几乎是必然的。他自己的做法是：

- 把 agent 跑在独立的 VPS 上，远离主开发机
- 给 AI 单独的 API key，便于审计
- Docker 的新 sandbox 模式可以隔离文件系统
- 自研的小工具 lockbox 用于在 AI 读到不可信 token 之后立刻切断文件系统访问
- 默认不使用 `--dangerously-skip-permissions`

他特别点名 OpenClaw 默认配置不安全，提醒希望用类似工具的人留意。

## "认知债务"与不肯放弃的最后一道关

观众问他是否真的让 loop 自己关闭项目。Parsons 回答说不会：他保留把任务标记为"完成"的最终权力，因为他担心一种他称作 **cognitive debt** 的状态——团队跑着跑着就再也搞不清自己代码库里有什么。同样的理由让他坚持读关键改动的 diff，尤其是任何涉及安全的部分：

> "我不能接受让 AI 把客户的数据弄丢。"

但他对未来留了一个口子：如果哪天 Mythos 这一代模型真的可信，他愿意把这一关也交出去。

## 真正的瓶颈通常不在 AI

工作坊接近尾声时，有人问如何把 Ralph Loop 推广到三人以上的团队。Parsons 的回答最后落到了 1984 年 Eli Goldratt 的《The Goal》和**约束理论**上：

- 系统总有一个最大瓶颈
- 不去解决那个瓶颈，其他优化都没意义甚至有害
- AI 工具往往会**暴露**瓶颈：如果发布流程是月度一次，每月 200 个 PR 一起上，AI 加速编码只会让团队跑得更慢
- 修好瓶颈，瓶颈会移动到别处，再去修新的瓶颈

他对并行多代理框架（Gas Town、MCP agent mail 等）持类似态度：在他看来，目前真正限制产出的不是每秒生成多少 token，而是人能多快地把需求表达清楚、再多快地审完结果。在这个瓶颈被解决之前，加 agent 解决不了核心问题。

## 留给观众的问题

整场工作坊给出的不是一个"如何写出更聪明的 prompt"的答案，而是一个让人不太舒服的工作伦理问题。Parsons 总结说，当 Ralph Loop 把可自动化的部分都吞掉之后，每个人都被迫认真区分两件事：哪些工作是自己**愿意**亲自做的，哪些只是因为暂时还不能信任 AI 才轮到自己头上。前者是真正的价值所在，后者迟早会被 loop 自己消化掉。

按照他在台上的说法：循环已经准备好了，问题是人想留在循环里的哪一段。
